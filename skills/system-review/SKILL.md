# システム設計レビューチェックリスト

## A. 要件との整合性（/5）

- [ ] アーキテクチャが機能要件を全てカバーしているか
- [ ] スループット・レスポンスタイム要件に対してスタックが妥当か
- [ ] MVPの範囲が過大でないか（スタートアップが Kubernetes は過剰など）
- [ ] 非機能要件（可用性・保持期間）の実現方法が明確か

## B. スケーラビリティ（/5）

- [ ] 単一障害点（SPOF）がないか（DB・キャッシュ・APIサーバー各レイヤーで確認）
- [ ] 水平スケーリング可能な設計か（ステートレス API か）
- [ ] 規模感（3年後）に対してボトルネックになるレイヤーが特定されているか
- [ ] DBコネクション数の上限を考慮しているか（コネクションプーリング）

## C. キャッシュ戦略（/5）

- [ ] キャッシュ対象・レイヤー・TTLが明示されているか
- [ ] TTLとデータ整合性のトレードオフが意識されているか
- [ ] キャッシュ無効化のタイミング・方法が具体的か
- [ ] キャッシュスタンピード（同時大量ミス）の対策があるか
- [ ] `KEYS` コマンド等の本番NGパターンを使っていないか

## D. DB設計（/5）

- [ ] 用途（OLTP / OLAP / KVS）に対して適切なDB種別が選ばれているか
- [ ] 読み取り重視ならリードレプリカ・マテビューを検討しているか
- [ ] 複数DB構成の場合、データ同期の整合性設計があるか
- [ ] インデックス設計がアクセスパターンに合致しているか
- [ ] マイグレーション・バックアップ・リカバリ計画があるか

## E. 計算基盤の配置（/5）

- [ ] 重い処理が同期APIに混在していないか（p95が要件を満たせるか）
- [ ] バッチ・非同期・リアルタイムの切り分けが適切か
- [ ] モデルトレーニング環境が本番インフラと分離されているか
- [ ] ワーカーの失敗時のリトライ・DLQ設計があるか
- [ ] キュー長のモニタリング・アラートが設計されているか

## F. セキュリティ（/5）

- [ ] 認証・認可の設計が具体的か（誰がどのリソースにアクセスできるか）
- [ ] 機密データの暗号化（転送中 TLS / 保存時）が考慮されているか
- [ ] SQLインジェクション・XSS・CSRF 対策が設計に含まれているか
- [ ] シークレット（APIキー・DB接続文字列）の管理方針が明確か
- [ ] レートリミットが設計されているか

## G. 運用・保守性（/5）

- [ ] ログ・メトリクス・分散トレーシングの設計があるか
- [ ] アラート条件（エラー率・レイテンシ・キュー長）が定義されているか
- [ ] デプロイ・ロールバック手順が考慮されているか
- [ ] 依存サービスが過多で障害時の切り分けが困難でないか

## H. コスト妥当性（/5）

- [ ] インフラコスト見積もりが現実的か
- [ ] 規模に対してオーバーエンジニアリングしていないか
- [ ] マネージドサービスとセルフホストのコストトレードオフを考慮しているか

---

## 判定基準

| 合計スコア | 判定 |
|-----------|------|
| 35-40 | ✅ 承認 |
| 25-34 | ⚠️ 修正後再レビュー |
| 0-24 | ❌ 要大幅修正 |

重大な問題（セキュリティホール・SPOF・計算基盤の根本的ミス）は1件でも ❌ とする。
